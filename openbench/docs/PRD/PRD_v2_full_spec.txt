Executive decision

OpenBench OS is not a single-surface demo app. It is a full product platform with three
primary surfaces and one shared backend:

   1. Bench Runtime Client - an installable desktop application for operators running
         procedures at the bench. Windows is the first-class packaged target in V1.

   2. Web Control Console - a browser-based application for protocol upload, review,
         publishing, run oversight, admin, and reporting.

   3. Tablet / Mobile Companion - a responsive installable web experience used mainly for
         photo capture, quick review, alerts, and handover consumption.

   4. Cloud Backend + AI Services - document ingestion, protocol compilation, retrieval,
         run orchestration, photo verification, reporting, audit logging, and integrations.

One-sentence product definition

OpenBench OS is a protocol execution runtime for lab work that compiles approved SOPs, SDSs,
equipment manuals, and lab rules into versioned execution graphs, then runs those graphs
with explicit state, checkpoints, photo verification, deviation capture, citations, and handover
reporting.

Product thesis

The product exists to solve one operational problem:
Approved lab procedures are hard to execute consistently in the moment of work.
The documents exist, but the runtime does not. Operators must translate dense SOPs, SDSs, and
manuals into action while handling equipment, maintaining sterility/cleanliness, wearing PPE,
and documenting deviations under time pressure. OpenBench OS provides the missing runtime
layer between approved documentation and real execution.

Product scope and success criteria

Product goals

V1 must:

   · turn uploaded, approved documents into a structured and reviewable protocol graph
   · publish immutable protocol versions
   · allow an operator to execute a published version through a dedicated run engine
   · persist run state outside the model
   · support step-level questions with citations
   · support note capture, deviations, timers, and confirmations
   · support photo-based verification against step-specific visual checklists
   · block or flag progress when required checkpoints are unresolved
   · generate structured handover reports from run events
   · provide manager/reviewer/admin workflows in a separate control console
   · support Windows-first bench deployment with the same core UI model available on the

         web
   · preserve auditability and source traceability for every major AI output
Non-goals for V1

The following are intentionally out of scope for the first shipping version:

   · direct instrument control
   · autonomous experiment design
   · open-ended protocol generation from scientific intent
   · real-time video monitoring
   · identity recognition of people in photos
   · biosafety or clinical decision support claims
   · formal 21 CFR Part 11 compliance claims
   · full ELN/LIMS parity
   · generalized inventory management
   · robotics orchestration

Success criteria

The product is successful when a lab can:

   · upload a real procedure pack and publish a reviewed protocol version
   · execute the procedure from a bench client without relying on PDF reading during the

         run
   · catch at least one meaningful setup mismatch or documentation gap before work

         proceeds
   · produce a structured run record and handover without reconstructing events after the

         fact
   · review deviations and unresolved checkpoints centrally
   · onboard new operators faster without sacrificing traceability

Platform strategy

Product shape in plain language

OpenBench OS is both:

   · a Windows-first installable bench application, and
   · a web product with admin/reviewer surfaces

It is not a choice between "Windows app" or "website". The shipping product is a multi-surface
platform with shared UI and shared backend.
Why this is the right shape

A web-only product is weak for bench execution because it under-commits to local runtime,
offline cache, and workstation deployment. A Windows-only native product is weak because
protocol management, admin, and remote review are naturally browser-friendly and easier to
deploy centrally.

The hybrid shape gives the product:

   · dedicated runtime UX for operators
   · centralized admin/review workflows
   · shared frontend code
   · deployability in mixed lab environments
   · a clean path to macOS/Linux later if demanded

Offline and network strategy

The Bench Runtime Client must tolerate poor or intermittent connectivity. V1 therefore
includes:

   · encrypted local cache of published protocol versions
   · local copy of active runs
   · local event queue
   · retry/sync service
   · read-only access to already-synced source snippets needed for active runs
   · upload retry for photos and audio if the network temporarily fails

The Control Console remains online-first.

Core user journeys

Journey 1 - Create and publish a runnable protocol

   1. Reviewer uploads SOP, SDS, equipment manual, and optional policy documents.
   2. System classifies, parses, chunks, and indexes source documents.
   3. Protocol Compiler generates a draft protocol graph, hazard map, and step-level visual

         checklist candidates.
   4. Reviewer inspects extraction confidence and edits any draft fields.
   5. Reviewer publishes a version.
   6. Published version becomes immutable and runnable.

Expected outputs
   · protocol version ID
   · normalized steps
   · prerequisites and blockers
   · required controls
   · visual checkpoints
   · citation map
   · review trail

Journey 2 - Start and execute a run

   1. Operator opens Bench Runtime Client.
   2. Operator selects a published protocol version.
3. System runs a preflight check: document completeness, local sync status, required
      acknowledgements, pending safety notices.

4. Operator starts run.
5. Run engine advances step by step with explicit current state.
6. Operator logs notes, starts timers, captures values, and completes steps.
7. System blocks or warns when required conditions are unresolved.

Expected outputs
   · run record
   · event history
   · timer events
   · confirmations
   · notes
   · deviations

   1. Operator reaches a step with visual checks.
   2. System displays only the relevant visual checklist for
   3. Operator uploads or captures a photo.
   4. Vision Checkpoint Engine returns:

            - confirmed items
            - missing/not visible items
            - cannot verify items
   5. Run engine updates blocker state.
   6. If unresolved critical checks remain, the step cannot

Journey 4 - Handle deviation or escalation

   1. Operator logs a deviation or the system flags a stop condition.
   2. Run transitions to warning, blocked, or escalated state depending on severity.
   3. Operator may:

            - correct and continue
            - request manager/reviewer decision
            - stop run
   4. All actions are appended as run events.

Journey 5 - Generate handover and review later

   1. Operator ends run or pauses for shift handoff.
   2. Report Generator builds handover from run events, notes, timers, measurements, and

         photo results.
   3. Manager or next-shift operator reviews handover.
   4. Report remains linked to exact protocol version and run record.

Summary

Map SDS and policy material to step-level safety requirements.

Responsibilities
   · parse SDS sections
   · extract PPE requirements
   · extract handling/storage/disposal constraints
   · identify first-aid/spill/fire notes
   · identify incompatibilities and escalation triggers
   · attach source citations to hazard rules
   · flag steps with missing SDS coverage

Module 4 - Run Engine

Summary

The Run Engine is the deterministic core of the product.

Responsibilities
   · create runs from published protocol versions
   · persist current step and progress
   · enforce prerequisites and blockers
   · manage timers
   · attach notes and measurements to steps
   · append all actions as events
   · support pause/resume/reopen
   · support override requests and outcomes

Design rule

The model is never the system of record for run state.
Run state lives in the database and local encrypted cache. The model reads state and returns
typed outputs.

Summary

Contextual Q&A scoped to the current step or approved source set.

   ·   distinguish between "confirmed by source", "inferred", and "not verifiable"
   ·   keep operator-facing answers terse by default

   ·    active
   ·    paused
   ·    blocked
   ·    awaiting_override
   ·    completed
   ·    cancelled
   ·    closed

Rules   a run enters preflight before active
   ·    a run enters blocked when a critical unresolved condition exists
   ·    a run enters awaiting_override when a user requests permission to proceed past a
   ·    critical blocker
        a run enters completed when all required terminal conditions are satisfied and the user
   ·    finalizes handover

   ·    archival workflows

   ·    in_progress
   ·    waiting_on_timer
   ·    waiting_on_checkpoint
   ·    blocked
   ·    completed

Completion rules

A step can be marked complete only when:

   · prerequisites are satisfied
   · required confirmations are recorded
   · required timers are satisfied or explicitly overridden
   · required visual checks are either confirmed or overridden with justification
   · no unresolved critical stop condition is active

Skip rules
   · only steps marked skippable in the protocol may be skipped by operators
   · all skips require reason capture
   · non-skippable step skip requests become override events

Deviation handling

Every deviation must capture:

   · run ID
   · step ID if relevant
   · title
   · description
   · severity
   · timestamp
   · actor
   · attachments if any
   · resolution state
   · whether supervisor review is required

Escalation handling

Escalations are triggered by:

   · source conflicts
   · missing required source coverage
   · unresolved critical visual mismatch
   · user request for unapproved substitution
   · hazard condition met
   · user-reported exposure/spill/fire/injury/malfunction
   · model unable to source answer for a critical instruction

Escalation outputs include:

   · escalation type
   · severity
   · user-facing message
   · required next action

AI system specification

AI responsibilities by pipeline

Pipeline A - Document understanding
Inputs: - uploaded documents - document metadata - parse quality signals
Outputs: - document type classification - normalized sections - chunk metadata - extraction
confidence

Pipeline B - Protocol compilation
Inputs: - SOP - SDS - equipment manual - optional policy overlays
Outputs: - protocol graph JSON - hazard map JSON - step visual checklist candidates -

Pipeline C - Step Q&A
Inputs: - current step - current run state - approved source snippets - user question
Outputs: - answer text - citations - confidence - escalation flag - suggested next UI action

Pipeline D - Vision checkpoint
Inputs: - current step - visual checklist - supporting source snippets - image(s)
Outputs: - checklist item results - rationale - evidence notes - overall status - recommended

Pipeline E - Report generation
Inputs: - run events - deviations - photo assessments - timers - notes - protocol metadata
Outputs: - handover report - manager summary - structured JSON report object -
PDF/HTML/Markdown render

Structured output requirement

All AI pipelines must return schema-validated JSON. Free-form text is allowed only in explicitly
defined fields such as answer_text or summary_paragraph.

Retrieval policy

   · retrieval scope defaults to the active protocol version and attached source pack
   · Q&A may include approved org-level policy overlays if configured
   · no retrieval from unrelated protocols unless the caller explicitly enters comparison

         mode and has permission
   · citations must include source document ID and page/section or chunk ID
Safety reviewer pass

Every operator-facing AI response that could influence action is passed through a lightweight
safety reviewer that checks:

   · unsupported claims
   · missing citations
   · overconfident phrasing
   · impermissible protocol invention
   · escalation requirements
   · forbidden identity/person analysis in photo responses

Confidence handling

Confidence is never used to silently suppress warnings. Instead:

   · low confidence extraction is highlighted for reviewer attention
   · low confidence photo results ask for another photo or manual confirmation
   · low confidence Q&A can answer with "not enough source support" and escalate to

         source view or reviewer

State model and event model

Why events matter

OpenBench OS is built around append-only event capture so that:

· the report is derived from actual recorded actions
· investigators can reconstruct what happened
· offline synchronization has a durable source of truth
· protocol analytics can be computed later
· the model is not responsible for remembering run history

Desktop client

   · Native capabilities: filesystem access boundaries, camera/mic bridge where needed,
         local notifications

Backend
   · API framework: FastAPI
   · Schema layer: Pydantic
   · ORM / migrations: SQLAlchemy + Alembic
   · Background jobs: Redis + Celery
   · Report rendering: HTML/Markdown to PDF pipeline

Data and storage
   · Primary database: PostgreSQL
   · Vector search: pgvector
   · Object storage: S3-compatible bucket storage
   · Tenant access control: Row-level access policies in the database plus application
         checks

Observability and operations
   · Error tracking: Sentry

   · Metrics/tracing: OpenTelemetry + metrics backend
   · CI/CD: GitHub Actions
   · Containers: Docker for all backend services

Why this stack is selected

Next.js App Router

Selected because it supports a shared React web surface for the Control Console and responsive
companion, while still allowing a coherent component model that can be reused inside the
desktop packaging approach.

Tauri 2 for desktop

Selected because it lets the product package a shared web UI into a dedicated desktop
application rather than requiring an entirely separate native UI stack.

FastAPI

Selected for typed Python APIs, straightforward background integration, JSON-schema-friendly
contracts, and proximity to document and AI processing code.
PostgreSQL + pgvector
Selected because the product needs transactional state, auditability, JSON support, and vector
retrieval in one system.
Redis + Celery
Selected because document parsing, compilation, photo assessment, and report generation are
asynchronous jobs that should not block request-response UX.
SQLite local cache
Selected because the operator app needs resilient local caching and event journaling for
intermittent network conditions.

Alternatives considered

Electron instead of Tauri
Rejected for V1 because the product benefits from a lighter desktop shell and a clearer
separation between shared web UI and minimal native wrapper.
Node/Express backend instead of FastAPI
Rejected because the most complex backend workloads - document parsing, OCR fallbacks,
retrieval orchestration, image assessment, and report generation - are already naturally
Python-centric.
Browser-only web app instead of packaged desktop client
Rejected because bench workflows need local caching, installability, controlled workstation
deployment, and a dedicated operator runtime.
Native Windows-only frontend
Rejected because it would slow development, fragment the UI codebase, and weaken the web
admin/reviewer surface.

Offline and synchronization design

Scope

Offline support is required for the Bench Runtime Client, not for every Control Console feature.
What works offline

   · open cached assigned protocols
   · resume active cached runs
   · read step instructions and citations already synced locally
   · capture notes, deviations, timer events, and confirmations
   · capture photos for later upload
   · queue override requests
   · generate provisional local handover preview from local event history

What does not work fully offline

   · publishing new protocol versions
   · org admin changes
   · fetching uncached protocol history
   · sending notifications to remote users
   · reviewer decisions that require server authority

Sync model

Local event journal

Every client-side mutation is first written locally with:

   · local event ID
   · local sequence number
   · idempotency key
   · pending/synced/error status

Server reconciliation

When connectivity exists:

   · queued events are posted in order
   · server acknowledges and maps them to canonical records
   · attachments are uploaded via signed URLs
   · any rejected events are marked with reason and surfaced to the operator

Conflict handling

Conflict resolution follows these rules:

   · published protocol versions never conflict because they are immutable
   · run events are append-only; two valid events can coexist
   · conflicting step outcomes become a reviewable inconsistency instead of silent overwrite
   · last-write-wins is allowed only for non-critical local UI preferences, never for run

         history
Security, privacy, and compliance posture

Security goals

   · protect proprietary procedures and attachments
   · prevent cross-tenant access
   · preserve auditability
   · reduce prompt injection risk from uploaded documents
   · protect operator credentials and local caches
   · avoid unsupported safety claims

Authentication and authorization

V1 includes:

   · email/password or SSO-ready auth provider integration
   · per-org memberships
   · role-based access control
   · database-enforced row isolation
   · short-lived access tokens with refresh flow
   · signed upload/download URLs for objects

Data protection

   · TLS in transit
   · encryption at rest for database and object storage
   · encrypted local SQLite database for the desktop cache
   · secrets stored in a dedicated secrets manager per environment
   · configurable retention and deletion windows

Privacy promises

   · customer documents are not used to train foundation models
   · data export and deletion workflows exist
   · attachments and reports inherit tenant access control
   · optional redaction support for names/sample IDs in exports later

Audit requirements

Audit logs must capture at least:

   · who uploaded or changed a protocol draft
   · who published a version
   · who started or ended a run
   · who approved or denied an override
   · who changed org settings
   · who exported sensitive reports
Compliance posture

V1 is built to be compliance-aware, not compliance-certified. The product may borrow concepts
such as immutable histories, role checks, time-stamped actions, and record-linking, but it must
not market itself as formally compliant with regulated-record standards until validated
accordingly.

Accessibility specification

Accessibility principles

   · hands-busy workflows are a first-order use case
   · important actions must be accessible without precise pointer control
   · every spoken response must be available as visible text
   · color cannot be the only status signal
   · critical information must remain readable with gloves, distance, and time pressure in

         mind

V1 requirements

   · keyboard navigable primary workflows
   · high contrast theme
   · large target mode
   · readable text scaling
   · one-step-at-a-time mode
   · text-to-speech for current step
   · repeat-last-instruction action
   · "critical only" condensed view
   · captions/transcripts for voice features
   · semantic labels for assistive technology
   · accessible summary of photo-check results, not image-only feedback

Voice feature posture

Voice is an enhancement layer, not the sole control path. Every voice action must have an
equivalent touch/keyboard action.

Performance and reliability targets

Operator UX targets

   · open cached protocol: under 2 seconds
   · start run from already-synced protocol: under 3 seconds
   · step transition: under 1 second for local state update
   · Q&A response: p95 under 5 seconds
   · photo assessment response: p95 under 15 seconds
   · report generation: p95 under 30 seconds
   · local note/deviation capture: under 500 ms

Backend SLOs

   · API availability: 99.9% monthly target for hosted production
   · job success rate: 99% with retry policy for transient failures
   · durable event persistence before user-facing success on any mutation
   · no silent data loss on client reconnect

Error handling requirements

   · failed AI response cannot lose operator-entered data
   · photo upload failure must preserve local file and retry state
   · every hard failure must produce a human-readable recovery path
   · unresolved sync errors remain visible until cleared

Risks and mitigations

Risk 1 - The product feels like an LLM wrapper

Mitigation: make the protocol graph, run engine, blockers, event model, and immutable
versioning first-class product primitives. Chat is a surface, not the product.

Risk 2 - Unsafe or overconfident AI output

Mitigation: schema-validated outputs, mandatory citations, safety-review pass, and
refusal/escalation for unsupported instructions.

Risk 3 - Offline complexity delays delivery

Mitigation: ship bounded offline scope focused on assigned protocols and active runs, with
append-only local event journaling rather than broad local mirroring.

Risk 4 - Reviewer burden is too high

Mitigation: confidence scoring, change highlighting, reusable review presets, and import
normalization to reduce manual cleanup.

Risk 5 - Photo verification becomes unreliable

Mitigation: restrict checks to visible checklist items, avoid hidden-state inference, add retake
flow, and design explicit "cannot verify" states.

Risk 6 - Protocol versioning becomes messy

Mitigation: immutable published versions, clear supersession rules, and runs pinned to exact
versions.
Risk 7 - Multi-surface product creates design drift

Mitigation: one shared design system, one shared schema library, and common state
semantics across desktop/web.

Out of scope register

The following remain outside V1 even if technically possible:

   · direct control of lab instruments
   · human identity recognition or compliance judgment from images
   · protocol generation from a scientific goal prompt
   · always-on video analysis
   · replacing local lab emergency procedures
   · medical or clinical decision support
   · certification of safety or regulatory compliance
   · general inventory/warehouse management

Open questions for kickoff

These questions should be resolved early in delivery planning:

   1. Which customer segment is the first beachhead: academic labs, biotech R&D teams, or
         industrial QC labs?

   2. Is Windows-first sufficient for the first 2-3 pilot customers, or is macOS packaging
         required on day one?

   3. Which override categories can managers approve versus safety reviewers?
   4. What retention defaults are acceptable for attachments and photo evidence?
   5. Which integration path matters first: webhook-only, CSV export, or direct ELN push?
   6. What minimum offline duration must the desktop client support without re-

         authentication?
   7. Which model providers are allowed in hosted versus self-hosted deployments?

Final product summary

OpenBench OS V1 is a full application platform, not a demo. It consists of:

   · a Windows-first installable Bench Runtime Client for operators
   · a browser-based Web Control Console for reviewers, managers, and admins
   · a responsive companion experience for tablet/mobile workflows
   · a cloud backend with document intelligence, run orchestration, photo verification,

         reporting, and audit services
The defining architectural choices are:

   · immutable published protocol versions
   · explicit run state machine
   · append-only event history
   · local desktop cache plus sync queue
   · AI outputs constrained by typed schemas and citations
   · photo verification scoped to step-specific checklists
   · reporting derived from events, not chat memory

This is the build-ready shape of the product.
